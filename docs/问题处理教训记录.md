## ❌ 犯的错误

### 错误 1：没有先验证就修复
```
❌ 错误流程：
用户反馈 → 直接修改代码 → 提交部署 → 用户验证

✅ 正确流程：
用户反馈 → 本地启动验证 → 截图确认 → 分析根因 → 修复 → 本地验证 → 提交部署
```

**后果**：
- 修复方向错误
- 浪费多次提交
- 用户反复验证

### 错误 2：没有截图确认
```
❌ 错误行为：
- 没有访问页面截图
- 没有查看控制台错误
- 凭想象修复问题

✅ 正确行为：
- 必须截图问题页面
- 必须检查控制台错误
- 基于证据分析根因
```

### 错误 3：没有逐步验证
```
❌ 错误做法：
- 一次修改多个文件
- 多个问题一起修复
- 最后才验证结果

✅ 正确做法：
- 一个问题一个问题修复
- 每次修改后立即验证
- 确保每个修复都有效
```

### 错误 4：没有遵守标准流程
```
❌ 实际执行：
接收问题 → 盲目修复 → 提交部署 → 用户说不对 → 再修复

✅ 标准流程：
接收问题 → 复现问题 → 分析排查 → 解决问题 → 本地验证 → 完整测试 → 提交部署 → 沉淀技能
```

### 第三次尝试（✅ 部分成功）
```
问题：代码高亮和 Mermaid 图表
行动：自定义 code 组件处理 Mermaid
结果：代码高亮✅ | Mermaid❌
教训：应该先截图确认问题范围

### 正确的做法应该是
```
1. 本地启动服务器
2. 访问 http://localhost:3000/knowledge/LLM/001_Transformer_示例
3. 截图页面
4. 打开控制台查看错误
5. 发现问题：
   - 代码块没有颜色 → 需要 rehype-highlight
   - Mermaid 显示代码 → 需要 Mermaid 渲染
6. 制定方案：
   - 安装 rehype-highlight
   - 安装 @theguild/remark-mermaid（需要 MDX，放弃）

7. 实施修复
8. 本地验证
9. 截图确认
10. 提交部署

## 🔑 关键教训

### 教训 1：验证优先
> **没有验证就没有发言权**

任何修复前必须：
1. 本地启动
2. 访问页面
3. 截图确认
4. 查看控制台

### 教训 2：小步快跑
> **一次只修复一个问题**

不要：
- ❌ 一次修改多个文件
- ❌ 一次修复多个问题
- ❌ 最后才验证结果

要：
- ✅ 一个问题一个问题修复
- ✅ 每次修改后立即验证
- ✅ 确保每个修复都有效

### 教训 3：证据驱动
> **基于证据，不靠猜测**

不要：
- ❌ 凭想象修复
- ❌ 盲目尝试
- ❌ 忽视控制台错误

要：
- ✅ 基于截图分析
- ✅ 基于错误日志
- ✅ 基于实际测试

### 教训 4：流程至上
> **流程是质量的保证**

标准流程不是束缚，而是保证：
- 不遗漏步骤
- 不跳过验证
- 不重复错误
